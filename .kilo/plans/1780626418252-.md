# Plan: Upgrade to Filament v4 Support

## Problem

This package currently targets `filament/filament:^3.0`. Filament v4 introduced several breaking
changes that break both the macro registration and the Blade view rendering. Additionally, v4 now
ships a native `->vertical()` method on `Filament\Schemas\Components\Tabs`, which conflicts with
the macro name this package uses.

## Branching Strategy

- **`main` (v3.x)** — Keep current state locked to `filament/filament:^3.0`, `php:^8.1`.
  Users who stay on v3 continue to benefit from the existing `->vertical()` macro.
- **`4.x`** — New branch targeting `filament/filament:^4.0`, `php:^8.2`. The macro is renamed to
  `verticalMobile()` and the Blade view is updated for the v4 Schema API.

On each branch the `composer.json` constraint pins the supported Filament version so that
`composer require afs19/filament-vertical-tabs` picks the correct branch automatically.

## Key v4 Breaking Changes That Affect This Package

### 1. Namespace unification (Forms + Infolists → Schemas)

| v3 | v4 |
|---|---|
| `Filament\Forms\Components\Tabs` | `Filament\Schemas\Components\Tabs` |
| `Filament\Forms\Components\Tabs\Tab` | `Filament\Schemas\Components\Tabs\Tab` |
| `Filament\Infolists\Components\Tabs` | `Filament\Schemas\Components\Tabs` (same class) |
| `Filament\Infolists\Components\Tabs\Tab` | `Filament\Schemas\Components\Tabs\Tab` |

This means we register **one macro** instead of two.

### 2. `getChildComponentContainer()` → `getChildSchema()`

The Blade view calls `$getChildComponentContainer()->getComponents()` to iterate tabs.
In v4 this is `$getChildSchema()->getComponents()` — the return type changed from
`ComponentContainer` to `Schema`.

**Migration:** change every occurrence in `vertical-tabs.blade.php`.

### 3. `vertical()` is a native method in v4 (conflict)

`Filament\Schemas\Components\Tabs` has a real `vertical()` method. Laravel's `Macroable::macro()`
cannot override a real method — PHP dispatches via method table, not `__call`.

**Action:** register the macro under a **different name** on the v4 branch.

### 4. PHP 8.2+, Laravel v11.28+ required

Bump `require.php` to `^8.2` in `4.x`.

### 5. Tailwind CSS v4 inside Filament

The custom Blade view uses Tailwind utility classes (`bg-primary-50`, `text-gray-900`, etc.).
Filament v4 moved its own classes to `@apply` in CSS files, so consumers need a custom theme with
a `@source` pointing to this package's `resources/views/`. The README should call this out.

### `src/FilamentVerticalTabsServiceProvider.php`

**Goal:** Register one macro (`verticalMobile`) instead of two (`vertical`) on
`Filament\Schemas\Components\Tabs`.

```diff
-use Filament\Forms\Components\Tabs as FormTabs;
-use Filament\Infolists\Components\Tabs as InfolistTabs;
+use Filament\Schemas\Components\Tabs;
```

Replace the two macro blocks with:

```php
Tabs::macro('verticalMobile', function (bool $isVertical = true) {
    /** @var \Filament\Schemas\Components\Tabs $this */
    $this->vertical($isVertical);

    if ($isVertical) {
        $this->view('filament-vertical-tabs::vertical-tabs');
    }

    return $this;
});
```

The macro:
1. Calls the native `->vertical($isVertical)` to set the internal flag (which adds
   the `fi-vertical` CSS class).
2. Overrides the view with the custom mobile-responsive Blade view.

**Why `verticalMobile`?** It signals that the macro provides the *mobile-responsive* UX
on top of the native vertical layout. The name is compact and suggests its purpose.

Users would write:
```php
Tabs::make('Settings')
    ->verticalMobile()
    ->tabs([...])
```

### `resources/views/vertical-tabs.blade.php`

**Goal:** Replace the v3 Schema API call with the v4 equivalent.

Two changes:

1. `$getChildComponentContainer()` → `$getChildSchema()`
2. The `$getChildSchema()` return type is `Schema`, but the `->getComponents()` API is
   the same, so no other Blade changes should be needed.

**Occurrences to replace (3 total in the file):**
- Line: `@foreach ($getChildComponentContainer()->getComponents() as $tab)` (×2 — mobile nav
  and desktop sidebar)
- Line: `@foreach ($getChildComponentContainer()->getComponents() as $tab)` (×1 — content area)

All three use the same pattern — iterate tabs. Replace each with:
```blade
@foreach ($getChildSchema()->getComponents() as $tab)
```

The rest of the Blade template (Alpine.js state, mobile panel, prev/next buttons, transition
classes) stays identical — it's a self-contained custom view that doesn't depend on internal
Filament rendering helpers beyond the tab list.
